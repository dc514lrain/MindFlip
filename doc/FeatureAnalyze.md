# 决策辅助工具小程序 - 项目需求与总体架构文档

## 1. 项目概述 (Project Overview)

### 1.1 项目背景与核心理念

本项目旨在打造一款兼具实用性与心理学深度的"决策辅助与复盘工具"。通过基础的随机工具帮助大众消解日常选择困难，通过高自由度的自定义决策引擎满足专业人士（如交易员、分析师）的严谨评估需求。

**核心产品 Aha Moment：** 揭示"抛硬币"背后的潜意识。系统不仅提供决策结果，更通过追踪用户"是否遵循了随机结果"，生成个人决策心理画像。

### 1.2 目标用户群体与分层

* **大众用户（Free 层）：** 需要解决日常繁琐选择（吃什么、去哪玩），渴望通过工具获得心理暗示与趣味性社交标签。
* **专业/高净值用户（VIP 订阅层）：** 股票交易员、商业分析师、项目经理等。需要将复杂的决策逻辑（风控、收益比、SWOT）抽象为可复用的标准化工具，并能在圈层内部分享。

### 1.3 VIP 订阅等级体系 (MVP / V1.0)

为保持 MVP 阶段的简洁性，采用 **两层制**，以"钩子策略"驱动转化：

| 层级 | 自定义工具上限 | 分享 | 核心权益 |
|------|:-----------:|------|---------|
| **Free** | 1 个 | ✅ (带尾标) | 全部基础工具（硬币/骰子/转盘/Pros&Cons）、统计分析、待决清单(Inbox)复盘、人格标签、订阅消息回访、**1 个自定义工具槽位（完整体验，含分享权）** |
| **VIP** | 5 个 | ✅ | Free 全部权益 + 5 个工具槽位、SWOT/PEST 等专业模板库、隐私模式、深度数据分析 |

**Free 的钩子策略：**
* Free 用户拥有 1 个自定义工具槽位，功能完全开放，可创建、使用、分享。
* 分享卡片带有尾标：*"由 [昵称] 使用决策大师创建"*（作为引流触点）。
* 当 Free 用户试图创建第 2 个工具时，触发订阅引导。

**VIP 的差异化价值：**
* 5 个槽位满足多场景需求（如：投资评估 + 穿搭决策 + 项目优先级 + ...）
* 深度隐私模式（端侧加密）
* 《个人决策行为周刊》深度解读

> **架构预留：** 后期可平滑拆分为 VIP 基础 (5 个) / VIP 进阶 (10+ 个) 两档，只需在权益配置层增加档位枚举，不影响现有数据模型。

### 1.4 订阅定价策略 (MVP)

采用 **双轨定价 + 锚点效应**：

| 方案 | 价格 | 定位 |
|------|------|------|
| 首月特惠 | ¥1.99 | 冷启动神器，极大降低决策门槛，培养使用依赖 |
| 月度订阅 | ¥15/月 | 入门锚点，通过略高月付凸显年付性价比 |
| 年度订阅 | ¥99/年 | **核心推荐。** 对于专业用户，¥99/年不到一杯咖啡的月均成本 |

> **支付接入：** 微信小程序虚拟支付，接入微信支付订阅扣费接口。

### 1.5 前端设计语言 (Editorial Design)

* **视觉风格：** 报纸/杂志风（Editorial Design）。强调排版、留白与网格系统。
* **色彩体系：** 以黑、白、高级灰为主基调（体现客观、冷静、专业），辅以高饱和度的单一点缀色（如克莱因蓝或荧光绿，用于强调"决策结果"和"动作按钮"）。
* **交互体验：** 大字体排版，动态排版卡片，数据分析报告生成时呈现为"个人专属杂志封面"的视觉效果，极具分享欲。

## 2. 核心功能需求 (Functional Requirements)

### 2.1 基础决策工具库 (Free)

#### 2.1.1 工具分类与交互策略

基础工具按交互模式分为两组：

**🔵 极速组：硬币、骰子**

* **零摩擦起步：** 点开即抛，瞬间出结果（正面/反面，数字 1-6），保留最纯粹的物理工具感。
* **事后轻备注：** 结果揭晓后，卡片下方淡入一行灰色可选输入框：
  > *"给这个决定加个备忘（选填）"*
* **Inbox 召回示例：**
  * 有备忘 → *"[抛硬币] 决定：火锅 (正面)"*
  * 无备忘 → *"[抛硬币] 结果：正面"*
* **设计逻辑：** 人类对自己在乎的潜意识冲突记忆力极强。如果用户自己都忘了这个正反对应什么，说明他根本不在乎——复盘时自然会选择"纯测试/好玩"。

**🟢 决策组：大转盘、优缺点对比 (Pros & Cons)**

* **前置输入：** 打开工具的第一步是输入选项。大转盘输入 N 个选项（如"火锅 / 日料 / 烤肉"），系统自动划分扇区。支持"历史选项一键填入"降低重复输入成本。
* **Inbox 召回示例：** *"[大转盘] 决定：烤肉"*

#### 2.1.2 触觉反馈

* 结合手机震动马达（Taptic Engine），在抛硬币、掷骰子时提供拟真的物理震动反馈。

#### 2.1.4 Inbox 收录边界总则

| 工具类型 | 是否进入 Inbox | 规则 |
|---------|:----------:|------|
| 硬币 / 骰子 | ✅ 自动 | 随机结果产生后自动进入 |
| 大转盘 | ✅ 自动 | 随机结果产生后自动进入 |
| 优缺点对比 (Pros & Cons) | 🔘 用户自选 | 结果页底部 Toggle：*"同步至待决清单进行复盘？"* |
| 自定义工具 (VIP) | 🔘 构建者配置 | 编辑态中为输出 Block 设置 `actionable: true`，则该工具运行结果自动进入 Inbox |

**自定义工具"可执行建议"标记：** 在搭积木编辑模式中，构建者可为输出 Block 开启"执行建议 (Actionable Advice)"开关。开启后，该工具运行时必须产出一个明确的"建议动作"（如：买入/卖出/观望）。运行完成后该建议动作自动录入 Inbox——VIP 工具从"计算器"进化为"指令型决策沙盘"。

#### 2.1.5 决策数据模型 (DecisionLog)

```typescript
interface DecisionLog {
  id: string
  user_id: string
  tool_type: "coin" | "dice" | "roulette" | "pros_cons" | "custom_schema"
  raw_result: string          // 原始物理结果: "heads", "4", "sector_2"
  semantic_result?: string    // 语义结果 (大转盘选项文本 / 自定义工具结论)
  user_memo?: string          // 事后轻备注 (极速组补充上下文)
  follow_status: "pending" | "followed" | "not_followed" | "expired"
  break_reason?: string       // 未遵循动机标签 (系统预设枚举)
  created_at: number          // 决策产生时间戳
  marked_at?: number          // 用户标记时间戳
}
```

### 2.2 高级自定义决策引擎 (VIP)

#### 2.2.1 "搭积木"式构建器

提供可视化组件库，VIP 用户可自由拖拽拼接决策工具。

* **创建入口：** 从模板克隆起步。用户选择一个内置模板（SWOT / 风险评估 / 空白模板），在此基础上增删组件。降低冷启动门槛，同时保留"从零搭建"的自由度。
* **组件关联方式：** 表单式配置。每个组件拥有配置面板，通过下拉选择"数据来源"来建立组件间关联。相比自由连线，表单式配置在移动端更精准可控，避免小屏幕误触。
* **输入组件：** 文本框、数值滑块、单选/多选标签。
* **逻辑组件：** 权重分配器（设置各项指标占比）、条件判断（如果 A>B 则触发 C）。
* **输出组件：** 雷达图、得分条、决策矩阵表。

**设计态 vs 使用态分离：**

| 维度 | 设计态（构建器） | 使用态（工具运行） |
|------|----------------|-------------------|
| 目标 | 灵活搭建 | 清晰使用 |
| UI 特征 | 组件面板、配置浮层、拖拽手柄 | 杂志内页排版，无"积木感" |
| 信息密度 | 高（展示所有可配置项） | 低（仅展示输入项与结果） |
| 交互 | 编辑、连线、参数调节 | 填写、滑动、点击运行 |

> **核心原则：** 使用界面不应暴露任何搭建痕迹。JSON Schema 需同时编码设计态配置与使用态布局信息——间距、字号层级、分组区块标题——使用户在使用时看到的是一张精心排版的决策工作纸，而非组件拼贴。

#### 2.2.2 自定义工具卡片管理

VIP 用户保存的自定义工具以 **"工具卡片"** 形态呈现，具备完整生命周期管理：

* **保存为卡片：** 构建完成后，生成一张工具卡片，包含工具名称、描述缩略图、分类标签。
* **二次修改：** 用户可随时编辑自己创建的工具卡片（修改组件配置、权重公式等）。
* **删除：** 支持删除不再需要的工具卡片。删除前二次确认。**历史决策记录与工具模板解耦存储**——删除工具只移除"入口"，过往决策记录完整保留在统计页和归档中。用户失去的是继续使用该工具的能力，而非过往的决策记忆。若该工具已被分享，已分享的副本不受影响（版本锁定，见 2.5）。
* **编辑 = 另存为新版本：** 用户编辑一个已有工具卡片时，实际操作是创建一个新版本工具（原工具保留）。系统提示：*"修改此工具将创建一个新版本，原版本将保留。是否继续？"* 这避免了因 Block 结构变化导致历史记录渲染异常，同时保护用户不丢失旧版本。
* **数量上限：** 按订阅等级限制可创建的工具卡片数量（Free 1 个，VIP 5 个）。
* **槽位满时引导：**
  * *Free 用户创建第 2 个工具：* 允许进入构建器完整搭建，在"保存"时才触发转化——*"你已经创建了一个很棒的工具。升级 VIP 解锁 5 个工具槽位，即可保存并使用它。"* 用户已投入搭建成本，此时转化率远高于冷弹窗。
  * *VIP 用户 5 个槽位满：* 提示——*"你的 5 个槽位已满。你可以删除一个不常用的旧工具，或关注即将推出的 Pro 版获取更多槽位。"* 不给压迫感，但埋下未来升级预期。
* **排序与收纳：** 支持拖拽排序、收藏置顶。

#### 2.2.3 工具卡片首页展示策略

**首页混排 + 视觉分界 (Editorial Index)：**

首页采用报纸版面分栏布局。上半部分为"极速物理工具"（硬币、骰子），下方以纯黑粗线分割出专栏区域：

> **"专属决策模型 (Pro Models)"**

| 用户类型 | 该区域展示内容 |
|---------|-------------|
| **VIP 用户** | 自己创建的工具卡片，精美排版，占据首页视觉重心 |
| **Free 用户** | 精美模板预览卡片（如"股票买入评估模型"、"MBTI 匹配度测试"）——所见即所得的诱惑力远大于干瘪的充值按钮 |

**Free 用户转化触点：** 点击模板预览卡片 → 弹出优雅的订阅引导：*"这是高级决策引擎，升级 VIP 即可保存并使用该模型，或创建你的专属工具"*

#### 2.2.4 工具使用态交互（iOS Shortcuts 模式）

借鉴 Apple Shortcuts 的"卡片作为发射器 (Launcher)"哲学，严格分离执行与编辑：

**点击卡片主体（主要热区）→ 拉起执行态：**
* 以半屏模态框 (Half-screen Modal) 承载使用表单——不跳页，保留空间让 JSON Schema 引擎渲染滑块和输入框。
* 填写完毕点击"运行"，结果展示并自动落入 Inbox。
* **状态记忆（解决重复使用痛点）：** 表单默认填充该用户上一次成功运行此工具时填入的参数。若参数未变，用户只需 *"点击卡片 → 点击运行"* 两步完成。

**点击卡片右上角 `···`（次要热区）→ 进入编辑态：**
* 进入构建器界面，可修改引擎逻辑、增删组件或生成分享码。

#### 2.2.5 专业模板库

内置 SWOT 分析、PEST 分析、投资风险收益比评估等高级模板，VIP 可直接套用或二次修改后保存为自己的工具卡片。

### 2.3 决策追踪与心理复盘闭环 (核心亮点)

#### 2.3.1 设计理念：待决清单 (Pending Inbox)

借鉴 GTD（Getting Things Done）理念，设计极简的"待决清单"作为决策追踪的唯一入口。整体数据流为：

```
工具使用 → 随机结果产生 → 自动进入待决清单 → 用户标记(遵循/未遵循+原因) → 归档至统计分析页
```

#### 2.3.2 待决清单 (Pending Inbox)

* **入口位置：** 小程序首页右上角或底部悬浮入口，带有未处理条数的小红点标识。低调但触达方便。
* **UI 风格：** 杂志报纸侧边栏风格——无边框纯文字排版，黑字白底，仅"遵循"和"未遵循"两个操作按钮带有强调色。用户像划掉待办事项一样快速处理。
* **列表内容：** 过去 48 小时内未标记的随机决策结果，以时间线形式排列。每条显示：
  * 工具类型图标（硬币/骰子/转盘）
  * 决策结果（如"正面 — 火锅"）
  * 产生时间（如"2 小时前"）
* **过期机制：** 超过 48 小时未标记的记录自动从待决清单移除，但仍保留在历史日志中（标记为"未反馈"）。
* **操作流程：**
  1. 用户点击一条记录
  2. 弹出极简操作浮层：两个主按钮（"遵循了"/"没遵循"）
  3. 若选择"没遵循"，展开系统预设的原因标签供选择（见 2.3.3）
  4. 确认后，该记录从待决清单消失，数据归档至统计分析

#### 2.3.3 未遵循原因标签体系 (系统预设)

当用户标记"未遵循"时，必须从以下系统预设标签中选择一个原因（不支持自定义）：

| 标签 | 含义 | 心理画像映射 |
|------|------|-------------|
| 🧠 直觉告诉我另一个更好 | 用户凭直觉推翻随机结果 | "直觉驱动型" |
| 🌧️ 外部条件变化了 | 客观环境改变导致无法执行 | "务实应变型" |
| 🔍 我只是试试看 | 用户本身就抱着测试心态 | "工具探索型" |
| 😅 结果我不喜欢 | 用户对随机结果不满意 | "结果导向型" |
| 🤔 我再想想 | 延迟决策，暂不行动 | "审慎观望型" |

> **设计原则：** 标签由系统预设而非用户自定义，保证数据一致性和心理画像的统计有效性。标签体系可在后续迭代中根据数据积累进行增补。

#### 2.3.4 双重回访机制

1. **主动回访：** 用户随时进入待决清单，快速标记。
2. **订阅消息唤醒（全部用户开放）：**
   * **聚合推送：** 24 小时内所有未标记决策合并为一条推送，而非逐条推送。文案示例：
     > *"你有 3 个决定等待复盘"*
   * **智能时间窗口：** 以决策生成后 24 小时为成熟期下限，实际推送时间控制在 **每日 9:00-21:00**。夜间产生的决策顺延至次日 9:00，避免深夜骚扰。
   * **点击跳转：** 用户点击通知 → 直接跳转待决清单 → 快速标记。
   * **过期静默：** 超过 48 小时仍未标记的记录不再推送，自动归档为"未反馈"。避免骚扰导致用户取消订阅授权。

> **设计哲学：** 微信生态下用户对通知容忍度极低（尤其是智能手表用户），聚合推送 + 智能窗口 = 克制的尊重。一次推送承载多条信息，制造"拆盲盒"的期待感而非骚扰感。

> **战略意义：** 订阅消息对 Free 用户全面开放。这不仅是功能，更是产品的核心生命线——留存触点与 VIP 转化入口。

#### 2.3.5 订阅消息授权流程

**首次授权时机：场景化情绪拦截**

用户使用完工具、结果揭晓后的瞬间是授权转化率最高的时刻。交互编排如下：

1. 结果卡片在屏幕上停留 2 秒
2. 卡片缩小并自动飞入右上角的 Inbox 图标（视觉暗示"已归档"）
3. 同时从底部升起半屏弹窗（Half-screen Modal），文案示例：
   > *"硬币已经给出了答案。24 小时后，我们会来检验你是否遵从了天意。允许系统为你投递一封复盘信吗？"*
4. 用户确认后拉起微信订阅消息原生授权弹窗

**被拒后的挽回策略：温和 + 克制**

* **Inbox Banner：** 待决清单顶部展示极简提示条（黑底白字）：
  > *"开启复盘送达服务，不再错过潜意识的回音 →"*
* **智能路由：** 点击 Banner 后判断授权状态——未被永久拒绝则重新拉起授权；已被永久拒绝则弹出引导图，指引用户通过右上角 `... → 设置` 手动开启。
* **三振出局：** 用户连续 3 次进入 Inbox 未点击 Banner，或主动点击 × 关闭后，Banner 永久隐藏。
* **设计哲学：** 高净值/高审美人群极其反感"牛皮癣"式营销。保持界面克制与纯净，赢得长期品牌信任。

### 2.4 数据分析与社交标签

#### 2.4.0 统计体系总览

| 层级 | 内容 | 位置 |
|------|------|------|
| **首页仪表盘 (Hook)** | 一行杂志风格大标题："本周你共进行了 24 次重要决策，理性遵循率 82%。本周人格：[绝对理性派]" | 首页顶部醒目位置，点击跳转复盘页 |
| **复盘页 (Review Tab)** | 上半部分 = 待决清单 (Inbox)；下半部分 = 全局统计 (Insights) | 底部导航栏第二 Tab |
| **单工具统计页** | 该工具专属的频次/分布/遵循率/历史列表 | 从各工具卡片进入 |

**底部导航栏 (Tab Bar) 结构 (MVP)：**

采用 **自定义 Tab Bar**（突破原生限制），中央放置快速启动按钮：

```
[ 决策 ]    [ 复盘 ]    [  ⚡  ]    [ 我的 ]
```

| Tab | 图标语义 | 内容 |
|-----|---------|------|
| **决策** | 🎯 | 首页：极速工具 + 专属决策模型卡片区 + 本周摘要 Hook |
| **复盘** | 📋 | 上半：待决清单 (Pending Inbox) / 下半：全局统计 (Insights) |
| **⚡ 快速决策** | ⚡ | **中央突出按钮。** 长按或点击拉起半屏面板，展示"最近使用的 3 个工具"+"全部工具"入口。紧急时刻一键触达，无需先回到首页。 |
| **我的** | 👤 | 个人中心：VIP 管理、人格标签海报、隐私设置、工具箱管理 |

**快速决策面板设计：**
* 半屏模态框 (Half-screen Modal)，从底部升起。
* 顶部 3 个最近使用工具的大图标（极速可达）。
* 底部"全部工具"入口——点击跳转首页决策区。
* 视觉处理：中央按钮在 Tab Bar 中微微凸起，使用点缀色（克莱因蓝/荧光绿），形成视觉锚点。

#### 2.4.1 复盘页设计 (Review Tab)

**上半部分 — 待决清单 (Pending Inbox)：**
* 待标记的决策记录时间线列表（详见 2.3.2）。
* 用户在此标记完成后，记录自动流向下方 Insights。

**下半部分 — 全局统计 (Insights)：**
* 总决策次数、总遵循率、工具偏好分布、人格标签（主 + 副）。
* 共享决策时间线（所有工具的历史记录聚合视图）。
* **核心体验：** "劳动即有即时反馈"——用户每处理完一条待决项，抬头就能看到数据变化。这种即时正反馈令人上瘾。

#### 2.4.2 工具统计页 (All Users)

每个基础工具拥有独立的统计分析页面，用户选择某个工具即进入该工具的统计视图：

* **使用频次：** 周期内的使用次数、时间分布热力图。
* **结果分布：** 如硬币正反面比例、骰子各面出现频率。
* **遵循率：** 该工具下的"遵循/未遵循/未反馈"比例，以及"未遵循原因"分布饼图。
* **历史记录列表：** 时间线展示所有历史决策及标记状态（数据来自待决清单归档）。

#### 2.4.3 决策心理学画像 (All Users)

根据用户的标记数据和工具使用习惯，系统生成具有设计感的 **"人格标签"**：

**标签体系：主标签 + 副标签 (1+N)**

| 层级 | 视觉权重 | 说明 |
|------|---------|------|
| **主标签** | 封面头条（大字号） | 用户最显著的行为倾向，极强的身份定义感，激发社交分享欲 |
| **副标签** | 副标题（小字号，最多 2 个） | 补充次要倾向，形成立体精准的心理画像，让用户产生"懂我"的 Aha Moment |

**标签类型：**

| 标签 | 判定条件 |
|------|---------|
| 🎯 绝对理性派 | 遵循率 > 90% |
| 🙏 天意顺从者 | 高遵循率 + 偏好硬币 |
| 🔄 逆向叛逆者 | 遵循率 < 30% |
| 🧠 直觉驱动型 | 高比例"直觉告诉我另一个更好" |
| 🧪 工具探索型 | 高比例"我只是试试看" |
| 🌧️ 务实应变型 | 高比例"外部条件变化了" |
| ⏳ 审慎观望型 | 高比例"我再想想" |

**计算周期：周刊仪式感**

* **结算周期：** 每周五下午 17:00（一周工作收尾、周末决策即将开始的节点）进行结算。
* **数据窗口：** 基于最近 30 天行为数据计算"本周人格"，同时保留"总人格"（全量历史）。
* **推送触达：** 期刊结算后推送订阅消息，文案示例：
  > *"《个人决策行为周刊》已出刊，你这周的直觉战胜了 80% 的硬币"*
* **设计意图：** 周期性期待感 + 周末社交分享窗口 → 打造高粘性 Habit Loop。

**冷启动预热机制：**

* **最低样本量：** 累计标记 10 次后解锁人格标签。
* **预热态 (前 9 次)：** 不显示"数据不足"，而是展示杂志排版风格的 _"潜意识解析中 (Analyzing...)"_ 状态。
* **进度引导：** 每次标记后给予反馈：
  > *"已捕捉到你的第 4 次心智反弹，还差 6 次即可生成专属人格档案"*
* **设计意图：** 利用"填满进度条"心理，将冷启动期转化为激活动力。

* **分享海报：** 支持生成杂志封面风格的人格画像海报（含主标签 + 副标签），分享至朋友圈。

#### 2.4.3 "我的"页面结构 (Me Tab)

采用杂志风个人档案布局，从上到下：

| 区域 | 内容 |
|------|------|
| **身份卡片** | 头像 + 昵称 + VIP 标识（或"升级 VIP"入口）+ 主标签大字展示 |
| **VIP 权益区** | VIP 用户：到期日、管理入口、续费按钮；Free 用户：*"解锁专业决策引擎"* 转化卡片 |
| **我的工具箱** | VIP 显示 `N/5` 槽位使用进度，入口进入编辑管理列表；Free 显示 `1/1` + 升级引导 |
| **人格海报** | *"生成我的决策人格周刊封面"* 入口，一键生成杂志风排版长图（含本周决策摘要 + 人格标签 + 统计数据 + 小程序二维码），天然适配朋友圈传播 |
| **数据导出** | 入口紧邻人格海报。MVP 阶段以杂志风长图导出为主（图片天然适合社交传播），后期可扩展 JSON/CSV 数据导出 |
| **设置区** | 隐私模式切换（标准/深度）、订阅消息管理、关于、反馈 |

#### 2.4.4 深度分析 (VIP)

结合行为经济学和决策心理学知识库，对 VIP 用户长期的决策趋势进行深度文本+图表解析（未来可接入 AI/LLM 生成）。

### 2.5 私域流转与分享机制 (VIP → VIP)

#### 2.5.1 工具分享码 (Share Code)

* VIP 用户可将自己创建的工具卡片打包，生成一段"决策口令"（类似淘口令）或微信分享卡片。
* 分享卡片展示工具的精美预览图，包含工具名称、描述和关键指标预览。

#### 2.5.2 版本锁定机制

**分享出去的永远是分享那一刻的版本快照。** 分享者后续对工具卡片的修改不影响已分享的副本。

* **分享者侧：** 可以持续迭代自己的工具卡片，每次分享生成独立的版本快照。
* **接收者侧：** 保存的是接收时刻的版本。接收者可以在此基础上二次修改，形成自己的分支版本。
* **设计意图：** 保护分享者的迭代自由，同时让接收者获得稳定可用的工具。

#### 2.5.3 接收端校验与增长裂变

* **VIP 用户：** 解析并保存分享的工具，存入自己的工具卡片库。
* **Free 用户：** 点击分享链接/复制口令打开小程序 → 展示工具的精美预览图 → 提示"开通 VIP 即可解锁专业级决策模型" → 核心增长裂变触点。

## 3. 总体技术架构设计 (Technical Architecture)

### 3.1 总体分层架构

* **表现层 (Client)：** 微信小程序原生框架 / 跨平台框架 (如 Taro 或 Uni-app，强烈建议使用跨端框架以兼顾未来 App/Web 的需求)。
* **网关层 (Gateway)：** 统一鉴权、限流、微信接口代理（订阅消息、支付验证）。
* **服务层 (Microservices/Modules)：**
  * *用户中心：* 账号体系、VIP 权益管理、微信 OpenID/UnionID 映射。
  * *引擎中心 (核心)：* JSON Schema 渲染引擎（负责将 VIP 搭积木的配置数据解析为前端 UI 界面）。
  * *决策数据流：* 记录用户的输入、计算过程及最终结果；管理待决清单的生成、标记与归档。
  * *打卡与调度：* 定时任务系统，触发微信订阅消息推送（24h/48h 回访）。
  * *分享服务：* 工具卡片版本快照的打包、加密与解析。
* **数据层 (Storage)：**
  * 关系型数据库 (MySQL/PostgreSQL)：存储用户基础信息、订单记录、关系映射、VIP 权益状态。
  * 文档型数据库 (MongoDB)：**极其重要**，用于存储结构灵活多变的"自定义决策工具模板 (JSON)"、"工具版本快照"和"非标准化的决策记录"。
  * 缓存 (Redis)：用于高频查询的标签、模板缓存及排行榜。

### 3.2 账号与跨端演进方案

* **初期阶段：** 采用 `微信一键登录`。
* **数据结构防腐：** 数据库设计时，必须记录微信的 `UnionID`（而非仅仅是 `OpenID`）。这样在未来推出独立的 App 或 Web 端时，只要用户通过同一微信授权，即可实现多端数据互通；并预留手机号绑定字段，为多端登录做准备。

### 3.3 核心技术方案：Block Protocol 动态引擎

为实现高级自定义功能，前后端不直接交互 UI 代码，而是基于 **Block Protocol 风格的 JSON Schema** 协议。

#### 3.3.1 Block Protocol 核心概念

每个组件是一个自包含的 **Block**，拥有三个标准接口：

```
Block {
  id: string           // 唯一标识，如 "b1"
  type: string         // 组件类型，如 "slider" / "weighted_score" / "output_gauge"
  config: object       // 组件自身的配置参数（如滑块范围、标签文案）
  inputs: {            // 输入槽位，值引用其他 Block 的输出
    "param_a": "b1.value"     // 引用 b1 的输出
    "param_b": "b3.score"     // 引用 b3 的 score 字段
  }
  position: {          // 使用态布局信息
    section: string    // 所属分组
    order: number      // 组内排序
  }
}
```

**为什么选择 Block Protocol：**
1. **完美映射移动端配置交互：** 在配置面板中，用户通过下拉菜单选择数据源——`block_id.output_key` 的引用结构天然匹配这种交互。
2. **逻辑扩展性强：** 未来可引入 `condition_gate`（条件门）、`api_fetch`（外部数据）等高级 Block 类型，底层协议无需变更。
3. **渲染层高度组件化：** 前端遇到 `type: "slider"` 渲染滑块组件，遇到 `type: "weighted_score"` 执行计算——一一对应，职责清晰。

#### 3.3.2 组件间引用规范

Block 之间的数据流通过 **点号路径引用** 建立：

```
<block_id>.<output_key>

示例：
  "b1.value"       — 引用 Block b1 的主输出值
  "b3.score"       — 引用 Block b3 的 score 字段
  "b5.passed"      — 引用条件门 Block b5 的布尔判断结果
```

前端渲染引擎在运行时按拓扑序求解：先计算被依赖的 Block，将结果注入下游 Block 的 inputs，再计算下游。

#### 3.3.3 Block 运行时计算逻辑

**拓扑排序求解：** 前端渲染引擎在运行时对 Block 进行拓扑排序，按依赖顺序依次求解。被依赖的 Block 先计算，结果注入下游 Block 的 `inputs`，再计算下游。

**循环依赖检测（双重保护）：**
* **保存时静态校验：** 用户在构建器中配置关联时，引擎实时进行拓扑排序检查（DFS 判环）。发现环则高亮相关 Block 并提示：
  > *"检测到循环逻辑，无法完成评估"*
* **运行时兜底：** 防止极端异步场景（如未来引入的 API 扩展）导致的漏检，运行时捕获循环后给出优雅降级提示：
  > *"此工具的逻辑绕了回来，请检查组件间的连接顺序"*

**输入校验（混合模式）：**
* 每个输入 Block 在构建时可设置 `required: true/false`（默认选填）和 `defaultValue`。
* **选填且未填：** 输出 `null`。下游逻辑 Block 需内置容错（如将 `null` 视为 0 或跳过该权重项）。
* **必填且未填：** 运行按钮置灰，高亮未填项——强制显性引导，防止莫名其妙报错。

**容错原则：** 所有报错信息使用用户语言，杜绝技术术语（如"Circular Dependency"）。

#### 3.3.4 MVP 阶段的 Block 类型约束

**底层 Schema 一步到位（完整 Block Protocol），但上层开放的 Block 类型在 MVP 阶段严格受限：**

| Block 类型 | MVP | 说明 |
|-----------|:---:|------|
| `text_input` | ✅ | 文本输入 |
| `number_slider` | ✅ | 数值滑块 |
| `single_choice` | ✅ | 单选标签 |
| `multi_choice` | ✅ | 多选标签 |
| `weighted_score` | ✅ | 权重打分（多项加权求和） |
| `arithmetic` | ✅ | 基础四则运算 |
| `output_gauge` | ✅ | 结果仪表/得分条 |
| `output_radar` | ✅ | 雷达图 |
| `output_matrix` | ✅ | 决策矩阵表 |
| `output_text` | ✅ | 文本结论输出 |
| `condition_gate` | ❌ | 条件分支（If-Else），Phase 2 开放 |
| `api_fetch` | ❌ | 外部数据拉取，Phase 2+ 开放 |
| `loop` | ❌ | 循环迭代，按需评估 |

**MVP 布局约束：** 强制单列瀑布流（通过 `position.order` 排序），不支持多列栅格。

**模板驱动策略：** MVP 阶段鼓励用户通过修改内置模板的权重和参数来创建工具，而非从零拼装底层逻辑块。

#### 3.3.5 通用性设计原则

> **本引擎为通用决策框架，不对任何具体场景（如股票交易、医疗诊断、体育博彩）做特化定制。** 所有行业特定的逻辑应由用户通过 Block 组合自行构建。系统只提供"积木零件"，不预设"搭好的房子"。

### 3.4 核心数据流设计

#### 3.4.1 决策追踪数据流 (Pending Inbox 模式)

```
[用户使用工具] → [产生决策结果]
                      ↓
              [写入决策记录表]
              status = "pending"
                      ↓
              [待决清单 (Inbox) 可见]
              (仅展示近48h, status=pending)
                      ↓
         ┌─ 用户标记遵循? ─┐
         ↓                  ↓
    status="followed"   status="not_followed"
    reason_tag = null   reason_tag = <用户选择>
         ↓                  ↓
         └──────┬───────────┘
                ↓
        [归档至统计分析页]
        (不可再修改标记)
                ↓
        [定时任务: 24h后检查]
        若仍为pending → 推送订阅消息
        若48h后仍pending → status="expired"
```

#### 3.4.2 工具分享数据流 (版本锁定)

```
[VIP用户A] → [创建工具卡片 v1] → [分享] → [生成快照 snapshot_A_v1]
                                                  ↓
                        [VIP用户B] → [接收] → [保存为B的工具卡片(独立副本)]
                                                  ↓
                        [VIP用户B可二次修改] → [B的工具卡片 v1.1]
                        
[用户A] → [继续修改工具] → [工具卡片 v2]  (不影响snapshot_A_v1)
[用户A] → [再次分享] → [生成新快照 snapshot_A_v2]
```

## 4. 用户体验设计细则

### 4.1 新用户首次启动体验 (Onboarding)

**策略：直接可用 + 事后教育 (A + C 组合)**

**首次进入首页：**
* 极速工具（硬币/骰子/转盘）完全可用，无需任何前置操作。
* 下方"专属决策模型"区域展示精美模板预览卡片——既是视觉填充，也是 VIP 价值感知。
* 无弹窗、无强制引导。保持"即开即用"的工具属性。

**首次决策后的事后教育：**
* 用户完成第一次抛硬币后，结果揭晓。
* 卡片飞入 Inbox 的同时，底部升起轻浮层：
  > *"你已经迈出了第一步。24 小时后，我们会问你最终的决定。这，就是复盘的力量。"*
* 不打断操作，只是事后轻触达。用户在自己真实的决策结果上理解复盘机制，远比抽象引导页有效。

**人格标签预热态：**
* 前 9 次标记期间，Insights 区域展示 _"潜意识解析中 (Analyzing...)"_ 状态，配合进度引导文案。
* 累计 10 次标记后解锁完整人格标签。

## 5. 非功能性需求 (Non-Functional Requirements)

### 5.1 隐私与数据安全 (极其重要)

#### 4.1.1 隐私分层体系

提供两级隐私设置，平衡数据安全与产品体验：

| 模式 | 适用用户 | 工具模板存储 | 输入数据存储 | 决策记录存储 |
|------|---------|:----------:|:----------:|:----------:|
| **标准隐私** (默认) | Free & VIP | 云数据库明文 | 仅存元数据 | 云端，UnionID 绑定 |
| **深度隐私** (VIP 专属) | VIP | 端侧加密后上云（密文） | 仅存元数据 | 云端，UnionID 绑定 |

**标准隐私：** 数据正常云端存储，保证强租户隔离。数据绝不参与公开排行榜，不作为公共 AI 训练集。

**深度隐私：** 用户在开启时设置本地安全密码。核心 JSON Schema 数据在客户端加密为密文后再存入云数据库。云端只存储密文。用户换设备时需输入安全密码解密——密码永不上传。

#### 4.1.2 隐私作用域（三维度保护）

| 数据类别 | 保护级别 | 策略 |
|---------|:-------:|------|
| **工具模板** (JSON Schema + 权重公式) | 🔴 最高 | 深度隐私模式下端侧全量加密。开发者后台看到的是乱码。 |
| **使用输入** (股价、金额等瞬时参数) | 🟠 高 | **阅后即焚。** 写入 Inbox / 历史日志时仅记录元数据（如"进行了一次股票评估，最终得分 85"），不保存用户输入的原始数值。 |
| **决策记录** (遵循/未遵循 + 原因标签) | 🟡 中 | 行为画像依据，正常上云但仅与 UnionID 绑定。已剥离输入参数，仅剩脱敏的行为数据，用于生成《个人决策行为周刊》。 |

#### 4.1.3 存储架构与防腐层

* **云平台：** 采用微信云开发 (CloudBase)，其 JSON 文档型数据库与 Block Protocol 引擎天然匹配。
* **Repository Pattern（防腐层）：** 前端 UI 层不直接调用 `wx.setStorage` 或云 API。所有数据访问通过 `DataService` 抽象层（如 `DataService.saveDecision()`）。初期内部可实现为本地存储，后期仅修改 Service 层即可平滑迁移至云数据库，UI 与引擎零感知。

### 5.2 性能标准

* 小程序首屏加载时间（FCP） < 1.5s，确保"随用随开"的工具属性。
* 物理引擎（如抛硬币动画）需维持在 60fps，保证高端杂志风 UI 的丝滑质感。

### 5.3 架构可扩展性预留

* **VIP 等级扩展：** 权益配置设计为枚举/配置驱动，后期从两层拆分为多层时只需增加配置项，不涉及数据迁移。
* **心理工具扩展：** 预留"高级心理决策工具"模块接口。未来可引入专业心理学量表（如决策风格量表 GDMS、大五人格决策子维度等）作为核心竞争力功能。相关标签体系和画像算法设计时应保持维度可插拔。
* **AI/LLM 接入：** 深度分析模块预留 LLM API 调用接口，未来可实现"一句话自动生成专属决策模型"及"AI 决策报告解读"。

## 7. 待决问题 (Open Questions)

以下问题留待开发阶段细化，不影响整体架构设计：

1. 人格标签的具体计算公式与阈值（需结合实际数据调优）。
策略： 不硬编码算法，建立一个简单的权重分值系统。
方案： 定义 4 个核心维度，每个维度赋予一个基础分值：
遵循率 (遵循次数/总次数)
反思深度 (备注字数/频率)
决策速度 (从结果生成到标记的时间差)
工具广度 (使用不同工具的种类)
落地： MVP 阶段先设定简单的 IF-ELSE 规则（如：遵循率 < 30% 为“叛逆者”）。等积累了 500+ 用户数据后，再引入决策风格量表（GDMS）进行机器学习聚类，那时标签会精准得多。
2. 待决清单"48 小时过期"阈值是否做成可配置项。
策略： 硬编码 48 小时作为默认值，但数据库预留字段。
方案： 暂不需要做成用户可配置项，这会极大增加系统的复杂度和逻辑分支。在后端（CloudBase）设定一个常量 PENDING_EXPIRE_HOURS = 48，后续如果用户反馈“太长”或“太短”，只需修改一行配置代码即可。
3. 微信订阅消息模板的具体注册文案（需符合微信审核规范）。
策略： 遵守“服务通知”类目，避开“营销推送”。
方案： 必须使用“决策结果提醒”或“待办任务提醒”模板。
文案示例：“你有一项关于[决策名称]的决策记录等待复盘，点击完成标记。”
避坑： 切忌在模板中出现任何“优惠”、“活动”等词汇，否则微信审核会被驳回，甚至被封掉推送权限。
4. 微信支付商品 SKU 的具体配置（月付/年付/首月特惠三种商品）。
策略： 使用“支付后台配置 + 前端动态拉取”模式。
方案： 不要在代码中死循环判断价格。后端维护一个 sku_config 表：
sku_id: monthly_sub_01 (¥15.00)
sku_id: yearly_sub_01 (¥99.00)
sku_id: trial_sub_01 (¥1.99 - 仅限首单)
逻辑： 每次用户进入 VIP 购买页，前端向后端请求当前可用的 SKU 列表，这样你可以在后台随时调整价格或关闭首月特惠，而无需发版更新小程序。
5. 人格标签类型可基于专业心理学文献扩展（如引入 GDMS 决策风格量表维度）。
暂无
6. 隐私模式端侧加密的具体算法选型（AES-GCM 等）。
策略： AES-256-GCM 是工业界标准。
方案： 这是处理端侧加密的最佳选择。
AES： 对称加密，速度快。
GCM 模式： 提供了加密的同时进行“完整性校验”的能力，防止密文被篡改。
前端实现： 在小程序端使用 CryptoJS 库实现 AES-GCM 加解密。关键点： 一定要确保密钥（Key）派生于用户设置的 PIN，即“PBKDF2 密钥派生函数”，这样即便 PIN 很短，也能转化为高强度的密钥。
